# 找搭子需求定稿（最简实现版）

## 1. 文档定位

本文档是“找搭子”功能本轮开发的**唯一实现基线**。

它基于以下两份文档整理并定稿：

- `docs/04_29_找搭子功能说明.md`
- `docs/04_29_需求明确.md`

整理原则：

1. 保留已经明确的业务目标
2. 解决两份文档之间的冲突
3. 在未明确处采用**最简可实现**方案
4. 作为本轮前后端联调、接口设计、页面实现的唯一口径

---

## 2. 本轮目标

本轮要实现的是：

1. 用户在前端点击“找搭子”
2. 填写地点、日期范围、旅行偏好
3. 系统创建一条**实时匹配请求**
4. 系统在后台用“帖子池”方式进行 1 对 1 匹配
5. 前端以“实时匹配”的形态展示状态变化
6. 双方都同意后生成见面信息
7. 双方各自可发送一次备注
8. 到见面当天结束后，本次找搭子流程结束

本轮**不做**：

- 回访页面
- 未赴约举报
- 管理员审核
- 处罚封禁
- 聊天系统
- 多人组局
- 地图 SDK 深度接入
- WebSocket / SSE

---

## 3. 产品定位与总原则

### 3.1 总体定位

本轮仍然沿用现有后端“招募 / 申请”思路作为底层匹配模型，但前台体验改造成“实时匹配”。

也就是说：

1. 底层仍然存在“帖子池”
2. 用户发起找搭子时，会自动生成一条属于自己的隐藏帖子
3. 系统从帖子池中为用户挑选最合适的对象
4. 前端不再展示“列表式找帖子”，而是展示“正在实时寻找搭子”

### 3.2 最简实现原则

本轮优先保证：

1. 状态机清晰
2. 前后端接口稳定
3. 页面可联调
4. 旧功能不被破坏

本轮不追求：

1. 复杂匹配算法
2. 复杂通知系统
3. 跨页面全局实时弹窗
4. 完整历史归档能力

---

## 4. 与旧功能的关系

### 4.1 旧功能保留

当前已有旧接口和旧页面继续保留，不在本轮删除：

- 旧招募接口
- 旧联系记录接口
- 旧“我的招募帖 / 我的联系记录”页面

### 4.2 新功能不替换旧接口

本轮新增一套“实时匹配外观接口”：

- 旧接口继续可用
- 新前端主入口逐步迁移到新接口
- 不强制一次性替换全部旧逻辑

### 4.3 隐藏帖子规则

实时匹配生成的帖子属于**隐藏帖子**。

规则如下：

1. 隐藏帖子会进入匹配池
2. 隐藏帖子不会出现在旧的公开招募列表中
3. 隐藏帖子不会出现在旧的“我的招募帖”页中
4. 隐藏帖子只服务于本轮实时匹配功能

换句话说：

- 旧的公开招募流继续存在
- 新的实时匹配流使用单独的“隐藏帖子池”
- 两者互不干扰

## 5. 本轮最终业务决策

### 5.1 一次只允许一个进行中的找搭子请求

同一用户同一时间只允许有一条进行中的实时匹配请求。

进行中状态包括：

- `pending`
- `matched_waiting_decision`
- `matched_accepted`

### 5.2 覆盖旧请求规则

最简决策如下：

1. 如果当前状态是 `pending`，允许用户重新提交条件
2. 重新提交时，后端取消旧请求并新建新请求
3. 如果当前状态是 `matched_waiting_decision`，不允许覆盖
4. 如果当前状态是 `matched_accepted`，不允许再发起新请求

### 5.3 本轮不保留历史页

本轮前端不做历史记录页。

终态请求例如：

- `failed`
- `cancelled`
- `finished`

对前端来说统一视为“当前没有进行中的找搭子请求”，找搭子入口重新回到输入页。

## 6. 输入字段定义

## 6.1 表单字段

找搭子输入页保留以下字段：

1. `destination`：旅行地点
2. `travel_start_date`：旅行开始日期
3. `travel_end_date`：旅行结束日期
4. `preference_tags`：旅行偏好标签，多选
5. `preference_text`：自由补充说明，选填

### 6.2 本轮删除字段

本轮不再使用：

- `days`

### 6.3 日期规则

1. `travel_start_date` 必填
2. `travel_end_date` 必填
3. `travel_end_date >= travel_start_date`
4. 日期范围为闭区间，匹配时按“是否有重叠日期”判断

### 6.4 偏好标签枚举

本轮前后端统一使用以下标签：

```json
[
  "特种兵",
  "慢旅行",
  "拍照",
  "美食",
  "自然风光",
  "人文历史",
  "早起",
  "夜景"
]
```

说明：

1. 标签支持多选
2. 标签用于前端展示和后端匹配加权
3. `preference_text` 作为自由补充，不替代标签

## 7. 页面结构与视图

本轮只做一个主入口页：`找搭子页`。

这个页面内部根据状态切换不同视图，不额外拆独立详情页。

### 7.1 页面视图

页面视图分为以下 4 种：

1. 输入页
2. `pending` 匹配中页
3. `matched_waiting_decision` 待决定页
4. `matched_accepted` 已确认赴约页
5. `failed / cancelled / finished` 结束页

### 7.2 页面进入规则

用户点击“找搭子”后，前端必须先调用：

`GET /api/match/realtime/current`

然后按返回状态进入视图：

1. 无当前请求：进入输入页
2. `pending`：进入匹配中页
3. `matched_waiting_decision`：进入待决定页
4. `matched_accepted`：进入已确认赴约页
5. `failed / cancelled / finished`：进入结束提示页，并允许重新填写

前端不能只靠本地缓存判断当前状态，必须以后端返回为准。

## 8. 主流程

## 8.1 发起找搭子

1. 用户进入输入页
2. 填写地点、日期范围、偏好标签、自由文本
3. 点击“开始匹配”
4. 后端创建一条 `match_request`
5. 同时创建一条隐藏帖子进入匹配池
6. 页面切换到 `pending`

## 8.2 匹配中

`pending` 状态下：

1. 页面展示当前输入条件
2. 页面每 5 秒轮询一次当前状态
3. 如果找到候选对象，则切换到 `matched_waiting_decision`
4. 如果到截止时间仍未匹配成功，则变成 `failed`

## 8.3 出现候选对象

出现候选对象后：

1. 页面主体显示候选卡片
2. 首次进入该状态时，额外弹一次匹配弹窗
3. 弹窗关闭只关闭 UI，不改变后端状态

### 8.3.1 再次进入页面时的规则

最简决策如下：

1. 如果候选仍有效，页面主体继续展示候选
2. 已关闭过的弹窗，不自动再次弹出
3. 用户仍然可以在页面主体上执行“同意赴约 / 拒绝继续寻找”

### 8.3.2 为什么不采用“再次进入只显示仍在搜寻”

不采用该方案。

原因：

1. 会丢失候选对象的操作入口
2. 与“关闭弹窗不改变状态”原则冲突
3. 前后端状态会更难解释

因此本轮定稿为：

- 候选还在，就继续展示候选
- 只是弹窗不重复弹

## 8.4 用户拒绝候选

当用户点击“拒绝，继续寻找”后：

1. 当前候选状态变为 `rejected`
2. 当前请求重新回到 `pending`
3. 页面切回匹配中页

### 8.4.1 同一人是否会再次被推荐

本轮采用最简且可控的规则：

1. 同一 `match_request` 内，被拒绝或过期的同一对象**不再重复推荐**
2. 用户如果重新发起一条新的请求，则可以重新匹配到这个人

这样做的原因：

1. 避免重复推送同一人造成死循环
2. 简化前端体验
3. 简化匹配器实现

## 8.5 用户同意候选

本轮要求：**双方都同意后才生成最终 `pair`**。

### 8.5.1 A 先同意、B 未同意时

本轮最简决策如下：

1. 请求主状态仍然保持 `matched_waiting_decision`
2. 当前候选记录里标记：
   - `my_decision = accepted`
   - `peer_decision = pending`
3. 前端页面显示“已向对方发出赴约确认，等待对方回应”
4. 不生成最终 `pair`

### 8.5.2 一方先同意、另一方后拒绝时

最简规则：

1. 当前候选变为 `rejected`
2. 双方各自的请求都回到 `pending`
3. 双方继续参与下一轮匹配

### 8.5.3 双方都同意时

当双方都同意后：

1. 生成 `match_pair`
2. 双方请求状态都变为 `matched_accepted`
3. 后端生成见面时间和见面地点
4. 前端进入已确认赴约页

## 8.6 已确认赴约

在 `matched_accepted` 状态下：

页面展示：

1. 对方头像
2. 对方昵称
3. 见面时间
4. 见面地点
5. 我的备注
6. 对方备注

### 8.6.1 备注规则

本轮规则：

1. 每个用户只能发送一次备注
2. 备注发送后不可再次编辑
3. 备注发送成功后，对方可见

## 8.7 结束

匹配成功后，页面持续展示到见面当天结束。

见面当天结束后：

1. `match_pair.status = finished`
2. 对应请求也进入完成态
3. 前端再次进入找搭子页时，回到输入页

## 9. 匹配算法（最简实现）

本轮只做最简单的可解释匹配。

## 9.1 候选基础条件

候选对象必须满足：

1. 不能是自己
2. 对方也处于可匹配状态
3. 对方没有已确认的有效 `pair`
4. 地点规范化后完全一致
5. 日期范围至少有 1 天重叠
6. 对方不是当前请求已经拒绝 / 过期过的对象

## 9.2 评分规则

在满足基础条件后，按以下顺序排序：

1. 标签重合数量多的优先
2. 自由文本关键词命中更多的优先
3. 日期重叠天数更多的优先
4. 更早进入池子的请求优先

### 9.2.1 自由文本匹配规则

本轮不做复杂 NLP。

最简规则：

1. 将 `preference_text` 按常见标点和空格拆词
2. 去掉空字符串
3. 统计双方词项重合数
4. 仅作为加分项，不作为硬性门槛

## 9.3 没有候选时的行为

如果当前没有任何合适候选：

1. 请求保持 `pending`
2. 不伪造假匹配
3. 直到后续轮询时有新对象可匹配
4. 或直到到达截止时间变成 `failed`

---

## 10. 见面信息生成规则

## 10.1 见面时间

本轮固定规则：

1. `meet_time` 固定为 `travel_start_date` 当地时间上午 `10:00`
2. 统一返回 ISO 8601 时间字符串
3. 本轮不允许用户修改

示例：

`2026-05-01T10:00:00+08:00`

## 10.2 见面地点

本轮优先由 AI 生成一条安全公共地点文本。

要求：

1. 白天
2. 公共区域
3. 交通方便
4. 人流较多
5. 易辨认

### 10.2.1 AI 不可用时的回退

如果 AI 调用失败，则使用模板回退：

`{destination} 主城区地标广场 / 游客中心附近`

## 10.3 匹配摘要

匹配卡片中的 `match_summary` 也优先由后端基于标签和自由文本生成简短说明。

最简要求：

1. 1 句中文
2. 只强调共同偏好
3. 不超过 40 字

## 11. 页面交互细则

## 11.1 `pending` 页

展示：

1. 地点
2. 日期范围
3. 偏好标签
4. 自由文本
5. 状态文案“正在寻找你的灵魂旅伴”

交互：

1. 允许退出页面
2. 退出不取消请求
3. 再次进入时恢复到 `pending`
4. 提供“取消匹配”入口

## 11.2 `matched_waiting_decision` 页

展示：

1. 对方头像
2. 对方昵称
3. 匹配摘要
4. 推荐见面地点
5. 倒计时或到期时间
6. 操作按钮

按钮：

1. `同意赴约`
2. `拒绝，继续寻找`

## 11.3 弹窗逻辑

本轮最简规则：

1. 只有找搭子页内做轮询与弹窗
2. 不做全应用跨页面实时弹窗
3. 其他页面只通过通知页或再次进入找搭子页感知状态

说明：

这比“所有页面都做实时提示”更简单，也更适合本轮实现。

## 11.4 通知策略

本轮只做轻量通知记录，不做复杂推送。

后端可在以下时机写入通知：

1. 找到候选对象
2. 双方成功确认赴约

但前端不要求全局即时弹出。

## 11.5 已确认赴约页

展示：

1. 对方头像与昵称
2. 见面时间
3. 见面地点
4. 我的备注
5. 对方备注

交互：

1. 若我尚未发送备注，则展示备注输入框和发送按钮
2. 一旦发送成功，输入区变只读展示

## 11.6 结束态页面

当状态为：

- `failed`
- `cancelled`
- `finished`

页面展示结束提示，并提供：

1. `重新填写条件`
2. `返回`

## 14. 接口定稿

## 14.1 创建 / 覆盖实时请求

`POST /api/match/realtime`

请求体：

```json
{
  "destination": "大阪",
  "travel_start_date": "2026-05-01",
  "travel_end_date": "2026-05-04",
  "preference_tags": ["拍照", "美食"],
  "preference_text": "喜欢白天暴走，不想熬夜"
}
```

行为：

1. 若当前无进行中请求，则新建
2. 若当前有 `pending` 请求，则取消旧请求并新建
3. 若当前为 `matched_waiting_decision` 或 `matched_accepted`，返回业务错误

## 14.2 获取当前状态

`GET /api/match/realtime/current`

返回建议：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "active": true,
    "request_id": "req_001",
    "status": "matched_waiting_decision",
    "destination": "大阪",
    "travel_start_date": "2026-05-01",
    "travel_end_date": "2026-05-04",
    "preference_tags": ["拍照", "美食"],
    "preference_text": "喜欢白天暴走，不想熬夜",
    "match_deadline_at": "2026-04-29T00:00:00+08:00",
    "candidate": {
      "candidate_id": "cand_001",
      "peer_user_id": "user_002",
      "peer_nickname": "ryan",
      "peer_avatar": "https://...",
      "meeting_place_text": "梅田站中央广场",
      "match_summary": "你们都偏好拍照和白天出行",
      "decision_expires_at": "2026-04-28T14:30:00+08:00",
      "my_decision": "pending",
      "peer_decision": "pending"
    },
    "pair": null
  }
}
```

若无活动请求：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "active": false
  }
}
```

## 14.3 同意候选

## 14.5 发送备注

`POST /api/match/realtime/pair/{pairId}/remark`

请求体：

```json
{
  "remark": "我会背黑色双肩包，穿白色外套"
}
```

规则：

1. 一人只能发送一次
2. 已发送再次提交返回业务错误

## 14.6 取消匹配

`POST /api/match/realtime/{requestId}/cancel`

规则：

1. 仅 `pending` 状态允许取消
2. 取消后隐藏帖子同步失效

## 15. 定时任务

本轮仍然需要最基础的后端定时任务。

## 15.1 请求失败任务

检查条件：

- `match_request.status = pending`
- `now >= match_deadline_at`

处理：

1. 更新请求状态为 `failed`
2. 隐藏帖子失效

## 15.2 候选过期任务

检查条件：

- `match_candidate.status = pending_decision`
- `now >= decision_expires_at`

处理：

1. 候选状态改为 `expired`
2. 对应请求回到 `pending`

## 15.3 已确认匹配结束任务

检查条件：

- `match_pair.status = active`
- 当前时间已经超过 `meet_time` 当天结束

处理：

1. `match_pair.status = finished`
2. 对应请求状态改为 `finished`
3. 找搭子入口恢复为输入态

## 17. 文案定稿

### 17.1 状态文案

| 状态 | 文案 |
| --- | --- |
| `pending` | 正在寻找你的灵魂旅伴 |
| `matched_waiting_decision` | 已匹配上你的灵魂旅伴 |
| `matched_accepted` | 你们已经约好见面啦 |
| `failed` | 暂时没有找到合适的搭子 |
| `cancelled` | 你已取消本次匹配 |
| `finished` | 本次找搭子已结束 |

### 17.2 按钮文案

- `开始匹配`
- `取消匹配`
- `同意赴约`
- `拒绝，继续寻找`
- `发送备注`
- `重新填写条件`
