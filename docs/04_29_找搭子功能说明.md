# 找搭子功能说明（实时匹配版）

## 1. 文档目标

这份文档用于重新整理“找搭子”一期功能，统一前后端对产品流程、页面状态、接口设计和超时规则的理解。

本版以**实时匹配**为主，不再把“延迟招募”作为当前一期主流程。

目标包括：

1. 明确用户从“输入偏好”到“匹配成功/失败/过期”的完整链路
2. 明确实时匹配界面的前端状态和交互
3. 明确匹配卡片、关闭卡片、重新进入页面后的状态保持逻辑
4. 明确同意后见面地点、见面时间、备注输入框等后续流程
5. 明确后端数据模型、状态流转、接口与定时任务

---

## 2. 一期功能概述

“找搭子”用于帮助用户围绕一次旅行快速匹配同一时间、同一目的地、偏好相近的旅伴。

### 2.1 一期保留的输入条件

用户点击“找搭子”后，进入输入页，保留以下字段：

1. 旅行地点
2. 旅行时间
3. 旅行偏好

用户点击“开始匹配”后，不再进入招募列表，而是进入**实时匹配界面**。

---

## 3. 核心产品流程

## 3.1 主流程总览

1. 用户进入“找搭子”输入页
2. 输入旅行地点、旅行时间、旅行偏好
3. 点击“开始匹配”
4. 系统创建一条实时匹配请求
5. 前端进入“实时匹配界面”
6. 匹配成功前，界面处于 `pending`
7. 一旦系统匹配到用户：
   - 会弹出一张匹配卡片
   - 实时匹配界面也同步进入“已匹配到人”的状态
8. 用户可以：
   - 同意赴约
   - 拒绝并继续寻找
   - 先关闭弹窗不做选择
9. 若用户关闭弹窗但未处理：
   - 重新进入找搭子页面时，仍然显示这条匹配结果
   - 用户仍然可以在页面里同意或拒绝
10. 若用户同意：
    - 后端生成见面时间与见面地点
    - 页面下方显示见面信息
    - 同时出现备注输入框与发送按钮
11. 匹配成功后，该页面持续显示匹配结果，直到约定见面当天失效
12. 若在有效期内一直没有匹配成功：
    - 到旅行第一天前 48 小时时，系统自动结束匹配
    - 页面提示“未找到合适的搭子”
    - 重新回到输入状态

---

## 4. 前端页面与状态设计

## 4.1 页面列表

一期建议只做以下页面/视图：

1. 找搭子输入页
2. 实时匹配页
3. 匹配弹窗卡片
4. 匹配确认后的见面详情区（嵌在实时匹配页内）

说明：

- 一期不强制拆独立“匹配详情页”
- 可以先把“匹配结果 + 同意拒绝 + 见面信息 + 备注输入”都放在同一个实时匹配页面里

### 4.1.1 与当前已有“找搭子按钮”和输入页的接轨方式

当前前端已经有：

1. “找搭子”按钮入口
2. 输入旅行地点、旅行时间、旅行偏好的输入页

这两个入口本次都继续保留，不需要推翻重做。

新的接轨方式如下：

1. 用户点击当前已有的“找搭子”按钮
2. 进入当前已有的输入页
3. 输入页继续保留：
   - 旅行地点
   - 旅行时间
   - 旅行偏好
4. 点击“开始匹配”后：
   - 不再进入旧的结果列表或招募式页面
   - 而是进入新的“实时匹配页”
5. 用户下次再次点击“找搭子”按钮时：
   - 前端先调用后端接口查询当前匹配状态
   - 若没有进行中的匹配请求，则进入输入页
   - 若已有进行中的请求，则直接恢复到对应状态页面

也就是说：

- “找搭子按钮”仍然是唯一入口
- “输入页”继续保留
- 本次核心改造发生在“开始匹配之后”的状态流转与页面展示

---

## 4.2 找搭子输入页

### 4.2.1 表单字段

1. 旅行地点
2. 旅行时间
3. 旅行偏好

### 4.2.2 按钮

- `开始匹配`

### 4.2.3 提交结果

点击“开始匹配”后：

1. 创建实时匹配请求
2. 返回请求 ID
3. 前端跳转实时匹配页

---

## 4.3 实时匹配页

这是一期最重要的页面。

该页面至少有 4 种状态：

1. `pending`
2. `matched_waiting_decision`
3. `matched_accepted`
4. `expired_or_failed`

---

## 4.4 `pending` 状态

### 4.4.1 展示内容

展示文案建议：

- 标题：`正在寻找你的灵魂旅伴`
- 副标题：`请稍候，我们正在为你实时匹配目的地、时间与偏好都更契合的人`

展示字段：

1. 当前填写的旅行地点
2. 当前填写的旅行时间
3. 当前填写的旅行偏好
4. 当前状态：`匹配中`

### 4.4.2 页面行为

1. 用户可以退出该页面
2. 退出后请求仍然有效，不自动取消
3. 用户再次进入“找搭子”时：
   - 如果该请求仍然 `pending`
   - 仍回到这个 `pending` 页面

### 4.4.3 失效规则

如果在**旅行第一天前 48 小时**仍未匹配成功：

1. 请求自动过期
2. 页面变为 `expired_or_failed`
3. 提示：
   - `暂时没有找到合适的搭子`
4. 同时提供按钮：
   - `重新填写条件`
   - `取消匹配`

---

## 4.5 匹配成功后的状态

一旦系统匹配到一个对象，前端会出现两层反馈：

1. **实时匹配页本身**进入“已匹配到人”的状态
2. **系统弹出一张匹配卡片**

这两处展示内容必须一致。

---

## 4.6 匹配弹窗卡片

### 4.6.1 触发时机

当系统成功匹配到对方时，立即弹出一张卡片。

### 4.6.2 卡片展示内容

卡片中需要展示：

1. 对方头像
2. 对方 ID 或用户昵称
3. 匹配到的见面地点
4. 与你匹配的相关偏好摘要
5. 一句更有氛围感的主文案

推荐主文案示例：

- `已匹配上你的灵魂旅伴`
- `你的旅途同频者已出现`
- `找到和你一起出发的人了`

### 4.6.3 卡片操作按钮

1. `同意赴约`
2. `拒绝，继续寻找`
3. 右上角关闭按钮

### 4.6.4 关闭逻辑

用户点击关闭后：

1. 仅关闭弹窗
2. 不代表拒绝
3. 当前匹配结果继续保留
4. 回到找搭子页面时，仍能看到相同的匹配结果内容

---

## 4.7 实时匹配页的“已匹配到人”状态

### 4.7.1 页面展示内容

当页面进入 `matched_waiting_decision` 状态时，展示内容与弹窗一致，至少包括：

1. 对方头像
2. 对方 ID / 昵称
3. 见面地点
4. 相关偏好信息
5. 主文案：`已匹配上你的灵魂旅伴`
6. 按钮：
   - `同意赴约`
   - `拒绝，继续寻找`

### 4.7.2 状态保持规则

即使用户关闭了弹窗：

1. 再次打开找搭子界面时
2. 仍然应显示此状态
3. 用户仍然可以同意或拒绝

### 4.7.3 超时规则

如果该匹配结果在一定时间内没有被同意或拒绝，应自动失效。

一期建议：

- 该匹配邀请设置独立过期时间 `decision_expires_at`
- 例如：创建后 15 分钟过期

过期后：

1. 页面不再显示该匹配结果
2. 请求重新回到：
   - 继续匹配
   - 或彻底过期

具体建议：

- 若整体请求仍未到总截止时间，则重新回到 `pending`
- 若已接近旅行第一天前 48 小时，则直接标记失败

---

## 4.8 同意赴约后的页面状态

当用户点击 `同意赴约` 后，页面进入 `matched_accepted` 状态。

### 4.8.1 后端返回内容

后端应返回：

1. 见面时间
2. 见面地点
3. 匹配对象信息
4. 当前匹配状态

### 4.8.2 页面新增展示

在原本匹配成功内容下方新增：

1. 见面时间
2. 见面地点
3. 备注输入框
4. `发送` 按钮

备注输入框用途：

- 用户填写“如何认出我”“我会穿什么”“补充约定事项”等内容

### 4.8.3 页面持续时间

匹配成功且同意后，该页面一直保留这条匹配结果：

1. 用户重新进入找搭子页面时仍然看到
2. 直到约定见面当天失效

一期建议规则：

- `match_pair` 在 `meet_time` 当天结束后自动归档
- 找搭子页重新恢复为输入条件页

---

## 4.9 拒绝后的页面状态

当用户点击 `拒绝，继续寻找` 后：

1. 当前匹配对象失效
2. 系统继续寻找下一位候选人
3. 页面重新回到 `pending`

若后端规则要求双方都确认一次后再进入下一轮，也可以设计为：

1. 当前卡片关闭
2. 请求继续进入候选池

但一期建议尽量简单：

- 拒绝即回到 `pending`

---

## 4.10 匹配失败状态

当在总有效期内仍未找到合适对象时，页面显示：

- 标题：`这次暂时没有找到合适的搭子`
- 说明：`我们已经为你持续匹配到旅行开始前 48 小时，建议调整时间、地点或偏好后重新尝试`

按钮建议：

1. `重新填写条件`
2. `返回`

---

## 5. 后端业务规则

## 5.1 一期只保留实时匹配

一期聚焦：

1. 创建实时匹配请求
2. 进入 `pending`
3. 生成候选匹配结果
4. 用户同意 / 拒绝
5. 同意后生成见面地点与见面时间
6. 页面在见面当天前持续展示
7. 见面时间过后触发回访
8. 未赴约核实成立后封禁账号 1 个月

暂不重点做：

- 招募帖子流
- 多人组局
- 聊天系统

说明：

- “见面后回访”与“未赴约处罚”建议纳入一期
- 因为它们直接决定找搭子功能是否具备基本安全闭环

---

## 5.2 核心状态枚举

### 5.2.1 `match_request.status`

表示用户的找搭子请求主状态。

建议枚举：

- `pending`
- `matched_waiting_decision`
- `matched_accepted`
- `cancelled`
- `expired`
- `failed`

说明：

- `pending`：匹配中
- `matched_waiting_decision`：已匹配到对象，等待用户决定
- `matched_accepted`：双方已接受，进入赴约状态
- `cancelled`：用户主动取消
- `expired`：请求自然过期
- `failed`：到截止时间仍未匹配成功

### 5.2.2 `match_candidate.status`

表示一次候选匹配结果状态。

建议新增中间表 / 结果表来表示“一次匹配提案”：

- `pending_decision`
- `accepted`
- `rejected`
- `expired`

---

## 5.4 匹配与安全规则

## 5.4.1 基础匹配规则

候选对象需满足：

1. 不能是自己
2. 当前没有处于进行中的已确认匹配
3. 旅行地点一致或高度接近
4. 旅行日期一致或在允许误差范围内
5. 旅行偏好相似度达到阈值
6. 双方请求都仍为有效状态

## 5.4.2 候选匹配推荐地点

匹配成功后，系统需要给出一个推荐见面地点。

该地点要求：

1. 白天
2. 公共地点
3. 市中心或景区主城区附近
4. 人流较多
5. 便于辨认和到达
6. 避免偏僻、夜间、封闭、私人住宅场景

推荐来源可以是：

1. 后端规则模板
2. 后端调用 AI 生成
3. 地图服务 POI 召回后再做安全过滤

一期建议：

- 先由后端或 AI 返回一条安全地点文本，不必一开始就接地图 SDK 深度搜索

## 6. 后端接口建议

## 6.1 创建实时匹配请求

`POST /api/match/realtime`

请求体：

```json
{
  "destination": "大阪",
  "travel_date": "2026-05-01",
  "preference_text": "喜欢暴走、拍照、白天行程紧凑"
}
```

响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "request_id": 101,
    "status": "pending",
    "match_deadline_at": "2026-04-29T00:00:00+08:00"
  }
}
```

## 6.2 获取当前实时匹配状态

`GET /api/match/realtime/current`

## 6.7 回访与未赴约接口建议

### 6.7.1 获取待回访任务

`GET /api/match/feedback/me/pending`

用于前端查询当前用户是否有待填写的回访任务。

返回建议：

```json
{
  "code": 0,
  "message": "ok",
  "data": [
    {
      "pair_id": 7001,
      "peer_user_id": 88,
      "peer_nickname": "ryan",
      "peer_avatar": "https://...",
      "meet_time": "2026-05-01T10:00:00+08:00",
      "meet_location_text": "梅田站中央广场"
    }
  ]
}
```

### 6.7.2 提交回访结果

`POST /api/match/pair/{pairId}/feedback`

正常赴约：

```json
{
  "result": "met_ok",
  "remark": "对方准时到了，沟通顺畅"
}
```

未赴约：

```json
{
  "result": "no_show",
  "remark": "约定时间后一直未出现"
}
```

后端处理：

1. 写入 `match_feedback`
2. 如果结果是 `no_show`
   - 创建 `no_show_case`
   - 将该匹配记录标记为待核实

### 6.7.3 管理员审核未赴约

`POST /api/admin/match/no-show-case/{caseId}/review`

确认未赴约：

```json
{
  "result": "confirmed",
  "review_remark": "确认未赴约，执行 1 个月封禁"
}
```

驳回：

```json
{
  "result": "rejected",
  "review_remark": "证据不足，不处罚"
}
```

确认未赴约后的后端处理：

1. `no_show_case.status = confirmed`
2. 创建 `user_penalty`
3. `start_at = now`
4. `end_at = now + 30 days`

返回当前用户在找搭子中的主状态。

响应建议：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "request_id": 101,
    "status": "matched_waiting_decision",
    "destination": "大阪",
```

## 7. 前端交互建议

## 7.1 页面进入逻辑

用户点击“找搭子”后，前端先调用：

`GET /api/match/realtime/current`

根据返回状态决定进入哪个页面视图：

1. 无进行中请求：进入输入页
2. `pending`：进入等待匹配页
3. `matched_waiting_decision`：进入已匹配待选择页
4. `matched_accepted`：进入已确认赴约页
5. `failed` / `expired`：进入失败提示并允许重新填写

说明：

- 前端不能只靠本地状态判断
- 必须以后端当前状态为准

## 7.2 `pending` 页面刷新方式

建议：

1. 页面打开后轮询当前状态
2. 每 3 到 5 秒一次
3. 如果变成 `matched_waiting_decision`，立刻显示卡片

后续可升级为：

- WebSocket
- SSE

## 7.3 匹配卡片关闭逻辑

关闭只关闭 UI，不修改后端状态。

前端可在本地记一个：

- `candidatePopupDismissed = true`

但再次进入找搭子页时：

1. 仍要从后端取当前状态
2. 页面主体仍然展示待处理匹配结果

## 7.4 同意后备注区

同意后页面底部展示：

1. 见面时间
2. 见面地点
3. 备注输入框
4. 发送按钮

发送后：

1. 调用备注接口
2. 更新当前页面的 `my_remark`

## 7.5 回访页面放在哪里最方便

一期前端建议采用“双入口”方式：

1. **通知页入口**
   - 当见面时间过后，系统推送“请确认对方是否赴约”
   - 用户点击通知后进入回访页面
   - 这是最适合做触达的入口

2. **找搭子记录页入口**
   - 即使用户忽略通知，也可以在“找搭子”相关页面里看到待回访任务
   - 适合作为兜底入口

推荐实现：

1. 通知页里出现一条待回访卡片
2. 找搭子页面 / 我的搭子页面里增加“待回访”状态入口
3. 两边都跳转到同一个“回访页面”

### 7.5.1 回访页面展示内容

建议展示：

1. 对方头像
2. 对方昵称 / ID
3. 见面时间
4. 见面地点
5. 回访说明文案
6. 两个按钮：
   - `对方已赴约`
   - `对方未赴约`
7. 一个备注输入框
8. 一个提交按钮

说明：

- 如果用户选择“对方未赴约”，该备注将作为举报说明提交给后端
- 后端后续进入核实与封禁流程

## 8. 定时任务建议

本节所有“定时任务”均指**后端定时任务**，不由前端承担。

原因：

1. 前端可能被关闭、挂后台或断网，不可靠
2. 匹配过期、候选失效、见面结束后的回访触发都属于后端业务状态
3. 必须以后端写库结果作为唯一真实状态来源

前端职责只是：

1. 调接口读取当前状态
2. 在 `pending` 阶段轮询或订阅更新
3. 根据后端返回结果切换页面展示

## 8.1 请求失败任务

定时检查：

- `pending`
- 且 `now >= match_deadline_at`

处理：

1. 更新状态为 `failed`
2. 前端下次进入找搭子页时显示“未找到合适的搭子”

## 8.2 候选卡片过期任务

定时检查：

- `match_candidate.status = pending_decision`
- 且 `now >= decision_expires_at`

处理：

1. 更新该候选状态为 `expired`
2. 如果请求主状态仍有效：
   - 回到 `pending`
   - 继续匹配

## 8.3 已确认匹配过期任务

定时检查：

- `match_pair.status = active`
- 且 `meet_time` 已到当天结束

处理：

1. 将 `match_pair.status` 改为 `finished` 或 `expired`
2. 将对应 `match_request` 从已完成状态恢复为不可继续显示
3. 找搭子首页恢复为输入页

## 8.4 回访触发任务

定时检查：

- `match_pair.status = active`
- `now >= meet_time`
- 且尚未创建回访任务

处理：

1. 为双方各创建一条回访任务
2. 将 `match_pair.status` 更新为回访中状态
3. 向双方推送“请确认对方是否赴约”

建议：

- 不一定在见面时间一到就立刻触发
- 一期可以设置缓冲时间，例如见面时间后 2 小时再推送

## 8.5 封禁过期任务

定时检查：

- `user_penalty.status = active`
- `end_at <= now`

处理：

1. 将处罚状态更新为 `expired`
2. 恢复用户正常使用找搭子功能的权限

## 10. 前后端联调约定

## 10.1 时间字段

统一使用 ISO 8601：

`2026-05-01T10:00:00+08:00`

## 10.2 状态字段

前后端统一枚举，不要前端自行猜测中文状态。

## 10.3 候选与最终匹配区分

必须区分：

1. 候选匹配：`candidate`
2. 最终已确认匹配：`pair`

否则前端会很难处理：

- 关闭弹窗但未拒绝
- 已匹配但未同意
- 已同意且已生成见面信息

---

## 11. 一期默认假设

本版文档默认：

1. 找搭子只做 1 对 1
2. 只围绕旅行地点、旅行时间、旅行偏好做实时匹配
3. 一次只推送一个最优候选对象
4. 用户可以关闭匹配弹窗，但不能因此丢失当前候选
5. 候选有独立决策过期时间
6. 请求总有效期截止于旅行第一天前 48 小时
7. 同意后由后端返回安全的白天公共地点和见面时间
8. 匹配成功界面持续保留到约定见面当天
9. 见面时间过后会触发回访
10. 若核实确认对方未赴约，则封禁该账号 1 个月

## 12. 推荐开发顺序

1. 定义 `match_request`、`match_candidate`、`match_pair`
2. 先完成创建请求接口与查询当前状态接口
3. 完成 `pending -> matched_waiting_decision` 的状态流转
4. 完成匹配卡片与页面展示
5. 完成同意 / 拒绝接口
6. 完成同意后见面时间、见面地点返回
7. 完成备注输入与发送
8. 完成超时任务：
   - 候选过期
   - 总请求失败
   - 已确认匹配到当天失效
9. 完成回访接口与回访页面
10. 完成未赴约核实与 1 个月封禁逻辑
