# 找搭子功能说明

## 1. 功能定位

“找搭子”一共只有 3 个页面，不做单独的“发布招募页”。

正确流程是：

1. 用户点击首页“找搭子”
2. 先进入条件输入页
3. 输入：
   - 目的地
   - 出发日期
   - 停留天数
4. 点击“开始匹配”
5. 后端根据这三个条件返回推荐搭子
6. 前端展示推荐卡片列表
7. 如果没有合适的，用户点击右上角“发布”
8. 系统直接把当前已经输入好的条件发布成招募卡片

也就是说：

- `发布` 不再进入新页面
- `发布` 直接使用当前输入框里的内容

---

## 2. 页面数量

一共 3 个页面：

1. 找搭子页 `pages/match/index`
2. 我的招募页 `pages/my-recruitments/index`
3. 我的申请记录页 `pages/my-match-applications/index`

---

## 3. 页面一：找搭子页

### 页面定位

- 用户点击首页“找搭子”后进入的页面
- 首屏默认先展示条件输入区
- 页面风格保持当前设计，不改成公开广场首页

### 默认展示内容

页面初始状态就是一个表单，包含：

- 目的地
- 出发日期
- 停留天数
- “开始匹配”按钮

右上角有一个“发布”按钮

---

## 4. 找搭子页交互逻辑

### 4.1 开始匹配

用户输入：

- `destination`
- `startDate`
- `days`

点击“开始匹配”后：

1. 前端调用后端匹配接口
2. 后端根据这三个条件返回推荐搭子
3. 前端在当前页下方展示推荐结果列表

注意：

- 首屏不是直接展示推荐卡片
- 必须等用户点击“开始匹配”后，才展示后端返回的匹配结果

### 4.2 匹配结果展示

匹配结果展示在当前页下半部分，采用卡片列表形式。

每张卡片展示：

- 用户头像
- 用户昵称
- 头像 / 昵称可点击进入对方主页
- 目的地
- 出发日期
- 停留天数
- 申请按钮

### 4.3 推荐卡片交互

- 点击头像 / 昵称 -> 进入对方主页
- 点击“申请” -> 向对方发起申请

按钮状态建议：

- 未申请：`申请`
- 已申请未处理：`申请中`
- 对方已通过：`已通过`
- 对方已拒绝：`已拒绝`

---

## 5. 找搭子页右上角“发布”按钮

### 页面逻辑

右上角“发布”不跳新页面。

用户点击“发布”时：

1. 前端直接读取当前输入框里的：
   - 目的地
   - 出发日期
   - 停留天数
2. 调用发布招募接口
3. 后端生成一条新的招募帖
4. 发布成功后，这条招募进入“我的招募”

### 前置条件

如果用户还没填完整：

- 目的地为空
- 出发日期为空
- 停留天数为空

则前端不允许发布，并提示用户先填写完整信息。

### 发布成功后建议

第一版建议：

- 发布成功 toast 提示
- 然后跳转到“我的招募”页

也可以保留在当前页，只提示“已发布成功”。

---

## 6. 页面二：我的招募页

### 页面定位

- 展示当前用户自己发布过的所有招募卡片

### 卡片内容

每张卡片展示：

- 目的地
- 出发日期
- 停留天数
- 当前收到的申请数量
- 当前招募状态

### 页面交互

- 点击某张招募卡片 -> 查看该招募下收到的申请

### 招募详情展示

进入某条我的招募后，下方展示收到的申请列表。

申请列表也采用卡片形式。

每张申请卡片展示：

- 申请人头像
- 申请人昵称
- 头像 / 昵称可点击进入申请人主页
- 对应招募信息：
  - 目的地
  - 出发日期
  - 停留天数
- 当前状态
- 操作按钮：
  - 同意
  - 拒绝

### 状态处理

- 点击“同意” -> 状态变为 `已通过`
- 点击“拒绝” -> 状态变为 `已拒绝`
- 处理结果同步给申请人

---

## 7. 页面三：我的申请记录页

### 页面定位

- 用于展示当前用户主动申请过的所有招募帖

### 每张申请记录卡片展示

- 对方头像
- 对方昵称
- 头像 / 昵称可点击进入对方主页
- 对方招募帖对应的：
  - 目的地
  - 出发日期
  - 停留天数
- 当前申请状态

### 状态定义

- `申请中`
- `已通过`
- `已拒绝`

### 页面作用

- 用户持续查看自己的申请进度
- 不需要回原帖重复查看

---

## 8. 完整业务流程

### 流程 A：匹配推荐

1. 用户进入找搭子页
2. 输入目的地、出发日期、停留天数
3. 点击“开始匹配”
4. 后端返回符合条件的推荐搭子列表
5. 前端在当前页下方展示推荐卡片

### 流程 B：发布招募

1. 用户进入找搭子页
2. 输入目的地、出发日期、停留天数
3. 点击右上角“发布”
4. 前端直接把当前输入内容提交给后端
5. 后端创建招募帖
6. 发布成功后进入“我的招募”

### 流程 C：申请别人

1. 用户在匹配结果中点击“申请”
2. 系统创建申请记录
3. 状态默认为 `申请中`
4. 该记录出现在“我的申请记录”里
5. 同时该申请会出现在对方的“我的招募”里

### 流程 D：处理申请

1. 招募发布者进入“我的招募”
2. 查看某条招募下收到的申请列表
3. 点击“同意”或“拒绝”
4. 申请状态更新
5. 状态同步到申请人的“我的申请记录”

## 11. 页面跳转逻辑

### 11.1 页面关系

- 首页“找搭子” -> `pages/match/index`
- `pages/match/index` 右上角“发布” -> 直接调用发布接口，不跳新页面
- 发布成功后 -> `pages/my-recruitments/index`
- “我的”页 -> “我的招募” -> `pages/my-recruitments/index`
- “我的”页 -> “我的申请记录” -> `pages/my-match-applications/index`
- 匹配结果卡片点击头像 / 昵称 -> `pages/user-profile/index?userId=xxx`
- 我的招募里的申请卡片点击头像 / 昵称 -> `pages/user-profile/index?userId=xxx`
- 我的申请记录里的申请卡片点击头像 / 昵称 -> `pages/user-profile/index?userId=xxx`

### 11.2 发布后的跳转

用户在 `pages/match/index` 输入：

- `destination`
- `startDate`
- `days`

点击右上角“发布”后：

1. 前端调用“创建招募帖接口”
2. 创建成功后 toast 提示“已发布”
3. 立即跳转到 `pages/my-recruitments/index`
4. 用户可以在“我的招募”中看到刚刚发出去的招募帖

### 11.3 申请后的跳转

用户在匹配结果卡片点击“申请”后：

1. 前端调用“提交申请接口”
2. 当前卡片按钮状态切换为 `申请中`
3. 不强制跳页，默认停留在当前匹配列表页
4. 用户后续可在“我的申请记录”中查看处理进度

### 11.4 处理申请后的同步

当招募发布者在“我的招募”中处理申请后：

- 同意 -> 申请记录状态更新为 `已通过`
- 拒绝 -> 申请记录状态更新为 `已拒绝`

同步要求：

- 我的招募页中的申请卡片状态即时更新
- 申请人的“我的申请记录”中对应记录状态也必须同步更新

## 15. 后端实现逻辑说明

这一节是给后端直接看的，说明“找搭子”功能在服务端应该怎么理解和实现。

### 15.1 功能本质

这个功能本质上包含两类对象：

1. 招募帖
2. 招募申请记录

后端需要同时支持两件事：

- 根据用户输入条件返回推荐招募帖
- 允许用户把自己的条件发布为招募帖，并允许别人申请

也就是说，“找搭子”不是单纯的 AI 匹配，也不是单纯的发帖，它是：

- 条件输入
- 条件匹配推荐
- 可选发布招募
- 对招募发起申请
- 招募发布者审批申请

---

## 16. 后端建议数据模型

### 16.1 招募帖表 `match_recruitments`

建议字段：

- `id`
- `publisher_user_id`
- `destination`
- `start_date`
- `days`
- `status`
- `application_count`
- `created_at`
- `updated_at`

字段说明：

- `publisher_user_id`：发布人用户 ID
- `destination`：目的地
- `start_date`：出发日期
- `days`：停留天数
- `status`：招募状态
- `application_count`：收到的申请数量，可冗余也可实时统计

`status` 建议取值：

- `open`
- `closed`
- `expired`

### 16.2 申请记录表 `match_applications`

建议字段：

- `id`
- `recruit_id`
- `publisher_user_id`
- `applicant_user_id`
- `status`
- `created_at`
- `updated_at`

字段说明：

- `recruit_id`：关联的招募帖 ID
- `publisher_user_id`：招募发布者 ID
- `applicant_user_id`：申请人 ID
- `status`：申请状态

`status` 建议取值：

- `pending`
- `approved`
- `rejected`

## 17. 后端接口处理逻辑

### 17.1 开始匹配 `POST /api/matches/recommend`

前端传入：

- `destination`
- `startDate`
- `days`

后端职责：

1. 用这三个条件查询可匹配的招募帖
2. 排除当前用户自己发布的招募帖
3. 可选排除当前用户已经申请过的招募帖，或者保留但返回已申请状态
4. 返回推荐列表

推荐规则第一版建议：

- 目的地优先精确匹配
- 出发日期允许一定范围内接近
- 停留天数允许接近值匹配

第一版不必做太复杂推荐算法，先做规则匹配即可。

返回结果中建议直接附带：

- 发布人信息
- 招募信息
- 当前用户对该招募的申请状态

这样前端就能直接渲染卡片。

### 17.2 发布招募 `POST /api/matches/recruitments`

前端传入：

- `destination`
- `startDate`
- `days`

后端职责：

1. 校验参数完整性
2. 创建一条新的招募帖记录
3. 状态初始值设为 `open`
4. `application_count` 初始为 `0`
5. 返回新创建的招募帖

注意：

- 前端没有单独“发布招募页”
- 所以前端点击右上角“发布”时，就是把当前输入框里的内容直接提交
- 该接口创建成功后，这条招募帖必须能在“我的招募”中被查到

### 17.3 提交申请 `POST /api/matches/recruitments/{recruitId}/apply`

后端职责：

1. 检查该招募帖是否存在
2. 检查该招募帖是否属于当前用户自己
   - 自己不能申请自己
3. 检查该招募帖是否仍然可申请
   - `status` 必须为 `open`
4. 检查当前用户是否已申请过该招募
   - 防止重复申请
5. 创建一条申请记录
6. 状态初始值设为 `pending`
7. 招募帖的 `application_count + 1`
8. 返回申请结果

成功后，前端需要在两个地方看到结果：

- 当前用户的“我的申请记录”
- 对方的“我的招募 -> 收到的申请”

### 17.4 我的招募列表 `GET /api/my/recruitments`

后端职责：

1. 查询当前登录用户发布的所有招募帖
2. 返回列表
3. 每条记录应至少包含：
   - `destination`
   - `startDate`
   - `days`
   - `status`
   - `applicationCount`
   - `createdAt`

### 17.5 我的招募详情 `GET /api/my/recruitments/{recruitId}`

后端职责：

1. 校验这条招募帖是否属于当前登录用户
2. 返回该招募帖的基础信息
3. 返回该招募帖收到的申请列表

每条申请记录建议带上：

- `applicationId`
- `applicantUserId`
- `applicantNickname`
- `applicantAvatar`
- `destination`
- `startDate`
- `days`
- `status`

### 17.6 同意申请 `POST /api/my/recruitments/{recruitId}/applications/{applicationId}/approve`

后端职责：

1. 校验当前用户是否为该招募帖发布者
2. 校验该申请是否属于该招募帖
3. 校验该申请当前是否仍为 `pending`
4. 将状态更新为 `approved`

同步要求：

- 申请人在“我的申请记录”中看到这条状态变为 `已通过`
- 招募发布者自己的“我的招募”页里也看到状态更新

关于是否只能通过一个申请人：

- 这个需要产品最终拍板
- 如果第一版想简单，建议默认“可通过多个申请”
- 如果想做单人配对逻辑，则需要：
  - 当一个申请被通过后
  - 该招募可转为 `closed`
  - 其他 `pending` 申请自动改为 `rejected`

### 17.7 拒绝申请 `POST /api/my/recruitments/{recruitId}/applications/{applicationId}/reject`

后端职责：

1. 校验当前用户是否为该招募帖发布者
2. 校验该申请是否属于该招募帖
3. 校验该申请当前是否仍为 `pending`
4. 将状态更新为 `rejected`

同步要求：

- 申请人在“我的申请记录”中看到这条状态变为 `已拒绝`
- 招募发布者自己的“我的招募”页里也看到状态更新

### 17.8 我的申请记录 `GET /api/my/match-applications`

后端职责：

1. 查询当前登录用户主动发起过的所有申请
2. 按时间倒序返回
3. 每条记录应带上：
   - 对方用户信息
   - 对应招募帖信息
   - 当前申请状态

前端要靠这个接口持续查看处理进度，所以状态必须准确。

---

## 18. 后端状态流转建议

### 18.1 招募帖状态

建议状态：

- `open`：可申请
- `closed`：已关闭，不再接受申请
- `expired`：已过期

### 18.2 申请状态

建议状态：

- `pending`：申请中
- `approved`：已通过
- `rejected`：已拒绝

## 19. 后端实现注意事项

### 19.1 防重复申请

同一个用户对同一条招募帖，第一版建议只允许存在一条有效申请记录。

### 19.2 禁止自己申请自己

如果 `applicant_user_id == publisher_user_id`，后端应直接拦截。

### 19.3 发布成功后的可见性

新发布的招募帖需要立刻满足两个条件：

1. 当前用户“我的招募”里能查到
2. 匹配推荐逻辑中其他用户有机会被推荐到

### 19.4 状态同步

后端是最终状态源。

前端以下页面看到的状态都应基于后端返回：

- 找搭子匹配结果页
- 我的招募页
- 我的申请记录页

### 19.5 第一版建议简化点

为了尽快落地，后端第一版建议：

- 不做复杂推荐算法
- 不做聊天
- 不做实时消息推送
- 先保证：
  - 匹配推荐能返回结果
  - 发布招募能创建成功
  - 申请记录能写入
  - 审批结果能同步更新
