# Faraway 客户端前端开发文档

## 1. 文档定位

本文档不是讨论性方案，也不是方向建议，而是用于直接指导 `client_rebuild/` 前端开发落地的执行文档。

目标：

- 作为客户端前端开发唯一基线
- 可直接据此生成页面代码、状态管理、接口封装
- 约束目录结构、字段命名、页面行为、数据流

适用范围：

- Uni-app 客户端工程：`client_rebuild/`
- 参考设计项目：`faraway/`
- 后端前提：`FastAPI + SQLite -> MySQL/PostgreSQL + 统一业务接口`

---

## 2. 项目边界

### 2.1 客户端工程

正式开发目录：

- `client_rebuild/`

### 2.2 参考项目

仅作为视觉和交互参考：

- `faraway/`

参考内容包括：

- 页面布局
- 视觉风格
- 文案语气
- 页面跳转关系
- 功能模块划分

执行原则：

- 页面视觉、排版、氛围感优先参考 `faraway/`
- 技术实现、页面职责、数据流、目录结构一律以本文档为准
- 当 `faraway/` 现有实现与本文档冲突时，以本文档为准，只保留其设计表达

不继承内容：

- Google 登录实现
- Firebase Auth
- Firestore 数据读写
- 前端直连 Gemini

### 2.3 当前后端前提

- 服务端框架：`FastAPI`
- 数据库：`SQLite`
- 文件上传：后端上传接口或后端签发上传凭证
- AI：后端统一代理模型服务

---

## 3. 技术栈与工程要求

### 3.1 前端技术栈

- 框架：`Uni-app + Vue 3`
- 语言：`JavaScript`
- 状态管理：`Pinia`
- 请求：封装 `uni.request`
- 样式：`scss`

### 3.2 工程要求

- 页面代码统一使用 `script setup`
- 接口调用不得直接写在页面内，必须经过 `api/`
- 跨页共享状态必须进入 `stores/`
- token、用户信息、草稿、缓存统一通过 `utils/storage.js`
- 页面只处理展示和交互，不直接拼业务请求细节

### 3.3 命名规范

- 页面目录：小写短横线或业务英文名
- store：`useXxxStore`
- 接口文件：按业务模块命名
- 接口字段：统一使用后端返回字段，不在页面层做临时重命名

## 6. 页面行为定义

本节用于定义每个页面必须覆盖的功能，不允许只实现页面外壳。

## 6.1 登录页

### 页面目标

- 支持演示登录
- 预留账号密码登录入口
- 预留手机号登录入口
- 预留微信登录入口

### 页面元素

- Logo / 品牌标题
- 登录方式列表
- 用户协议提示

### 页面行为

- 演示登录：直接写入本地登录态并进入首页
- 账号密码登录：进入账号密码表单流程
- 手机号登录：进入手机号验证码流程
- 微信登录：本期只保留按钮与“暂未开放”提示，不接真实登录流程

### 页面状态

- `loading`
- `loginType`
- `form`
- `agreementChecked`
- `errorMessage`

### 必须覆盖的原功能

- 品牌标题展示
- 演示登录入口
- 手机号登录入口
- 微信登录入口
- 用户协议提示

### 新增补全要求

- 手机号登录需要完整表单流转
- 账号密码登录需要完整表单流转
- 登录失败需要明确错误提示
- 登录后统一进入首页并恢复用户信息

## 6.2 首页

### 页面目标

- 作为产品入口页
- 提供“做攻略”和“找搭子”主入口
- 保留 `faraway/` 式首页首屏结构
- 首页首屏以搜索框和两个核心按钮为主

### 页面元素

- 顶部品牌视觉
- 搜索框
- 两个核心 CTA：
  - 做攻略
  - 找搭子
- 语音输入图标占位
- 可选的次级推荐内容预览区

### 页面行为

- 点击做攻略进入 `strategy-form`
- 点击找搭子进入 `match`
- 点击搜索框进入统一搜索结果页
- 点击搜索提交后进入搜索结果页
- 首页首屏不承载小红书式信息流列表

### 必须覆盖的原功能

- 顶部沉浸式视觉
- “做攻略”入口
- “找搭子”入口
- 搜索框展示
- 搜索框右侧语音图标占位

### 新增补全要求

- 首页搜索需要形成真实跳转路径
- 首页 CTA 需要有空态和加载态逻辑
- 首页可保留少量推荐预览区，但不作为主结构

## 6.5 AI 做攻略页

### 页面目标

- 采集用户出行需求
- 请求后端 AI 接口生成攻略
- 支持保存草稿和正式保存

### 表单字段

- `destination`
- `days`
- `budget`
- `hotelRequirement`
- `allergies`
- `pace`
- `groupType`

### 页面行为

- 表单校验
- 调用 AI 生成接口
- 展示生成结果
- 支持“保存草稿”
- 支持“正式保存为攻略”
- 支持返回修改参数重生成

生成与保存链路补充：

- `AI 生成` 只负责返回一份“攻略结果草案”，默认不直接入库
- 用户点击“保存草稿”后，前端调用攻略草稿接口，内容保存到草稿箱
- 用户点击“正式保存为攻略”后，前端调用攻略发布接口，内容保存为正式攻略
- 正式保存成功后，跳转到 `pages/strategy-detail/index?id=新攻略ID`
- 正式保存成功后的攻略，必须同步出现在“我的发布”中
- 草稿内容不进入“我的发布”，只进入草稿箱

### 必须覆盖的原功能

- 分步或整页表单输入
- AI 结果展示
- 行程结果区块

### 新增补全要求

- 原 Web 端“生成后无法真正保存”必须补齐
- 结果页必须支持保存草稿
- 结果页必须支持正式入库
- 生成失败必须有重试按钮

## 13. 后端与数据库选型对前端的影响

本节用于回答：

- 后端是否使用 Firebase / uniCloud / 自建后端，会怎样影响前端
- 前端为了适配不同数据库方案，需要做什么约束
- 哪些地方前端可以无感，哪些地方前端必须跟着调整

## 13.1 基本原则

理想前端不直接依赖数据库品牌，只依赖：

- 登录方式
- 接口路径
- 请求参数
- 返回字段
- 鉴权方式
- 上传方式

也就是说：

- 数据库内部是 `SQLite`、`MySQL`、`PostgreSQL`、`Firestore` 还是 `uniCloud DB`
- 前端原则上都不应该直接关心

但前端一定会受到以下内容影响：

- 认证方案
- 接口契约
- 数据结构
- 分页方式
- 搜索能力
- 上传流程
- 实时能力是否存在

## 13.2 不同方案对前端的影响

### 方案 A：自建后端 + SQLite / MySQL / PostgreSQL

前端特点：

- 最通用
- 最清晰
- 最适合作为当前默认方案

前端需要适配的内容：

- 统一通过 HTTP API 调用
- 统一处理 token
- 统一处理分页字段
- 统一处理上传接口
- 统一处理 AI 接口

对前端的好处：

- 不和具体数据库耦合
- 页面逻辑最稳定
- 后续数据库从 SQLite 换 MySQL，前端通常无需大改

对前端的要求：

- 严格分层
- 不能在页面里写数据库思维的逻辑
- 不能把 SQL 表结构细节直接带进页面命名

### 方案 B：uniCloud

前端特点：

- 和 Uni-app 生态更近
- 开发速度更快
- 前后端可以更紧耦合

前端需要适配的内容：

- 可能会依赖 uniCloud 的调用方式
- 可能会依赖云函数 / 云对象调用风格
- 上传与登录能力可能使用平台内置方式

对前端的好处：

- 开发快
- 少写后端基础设施

对前端的风险：

- 更绑定 DCloud 生态
- 后续迁移到自建后端时，前端改动会变大

### 方案 C：Firebase / Firestore / Firebase Auth

前端特点：

- Web 项目里很顺
- 客户端场景不建议作为当前默认路线

前端需要适配的内容：

- Firebase SDK
- Firebase Auth 用户态
- Firestore 文档读写逻辑
- Google 生态网络与兼容问题

对前端的风险：

- 容易把数据库操作写进前端
- 容易把第三方服务细节写进页面
- 不适合作为当前客户端的长期主路径

## 13.3 哪些变化前端通常不用改

如果你前端已经严格遵守本开发文档分层，那么以下变化通常前端不需要大改：

- SQLite 升级为 MySQL
- MySQL 升级为 PostgreSQL
- 后端内部调整表结构但对外字段保持不变
- 后端更换 AI 模型但输出 JSON 不变
- 后端更换对象存储服务但上传接口保持不变

也就是说，只要接口契约不变，前端可以基本无感。

## 13.4 哪些变化前端一定要跟着改

以下变化会直接影响前端：

- 登录方式变化
- token 机制变化
- 返回字段变化
- 分页方式变化
- 搜索接口变化
- 上传流程变化
- 实时更新机制变化

例如：

- 如果登录从 Firebase Auth 改为后端 token
  - 前端要重写登录态管理
- 如果列表分页从 `cursor` 改为 `page/pageSize`
  - 前端列表逻辑要改
- 如果上传从“直接上传云存储”改为“先拿上传凭证再上传”
  - 前端上传服务要改

## 13.5 前端必须遵守的适配规则

为了降低未来切换数据库或后端方案时的成本，前端必须遵守以下规则：

### 规则 1：页面不得直接处理数据库逻辑

禁止：

- 页面中直接写 Firestore 查询
- 页面中直接写 uniCloud 数据库调用
- 页面中直接拼数据库筛选条件

必须：

- 页面只调 store 或 api

### 规则 2：接口模块统一抽象

例如：

- `api/modules/strategy.js`
- `api/modules/post.js`
- `api/modules/user.js`

无论后端是自建、uniCloud 还是别的方案，页面层都不应该知道。

### 规则 3：上传流程必须封装在 `services/upload.js`

不能把上传逻辑散落在页面里。

这样即使未来从：

- 本地 mock 上传
- 切换到 COS
- 切换到 OSS
- 切换到 uniCloud 存储

页面也不需要大改。

### 规则 4：登录逻辑必须封装在 `api/modules/auth.js` + `stores/user.js`

不能在多个页面分别处理：

- token 保存
- 登录态判断
- 用户资料恢复

### 规则 5：所有页面都只认统一字段

例如前端统一使用：

- `userInfo`
- `strategyList`
- `postList`
- `favoriteCount`
- `likeCount`

不允许页面里同时混用：

- Firebase 风格字段
- 自定义临时字段
- mock 字段名

## 13.6 当前项目的默认结论

当前项目默认采用：

- 前端：`Uni-app`
- 后端：`FastAPI`
- 数据库：`SQLite -> MySQL/PostgreSQL`

原因：

- 最适合当前“完整重做客户端并补齐原项目功能”的目标
- 最不容易和第三方平台强绑定
- 最方便后续补手机号登录、微信登录、上传、匹配与 AI 能力

### 对前端的直接要求

从现在开始，客户端代码必须满足：

- 不依赖 Firebase SDK
- 不依赖 Firestore 查询模式
- 不依赖 uniCloud 特有调用方式
- 统一按“HTTP API + token + 业务字段模型”开发

## 17. 页面跳转逻辑总表

本节用于约束客户端页面间跳转，不允许页面自行发散设计跳转路径。

## 16.1 全局跳转原则

- 未登录用户：
  - 只能进入登录页
  - 如果访问需要登录的页面，统一拦截并跳回登录页
- 已登录用户：
  - 默认进入首页
  - TabBar 页面之间统一使用 `switchTab`
  - 非 TabBar 页面统一使用 `navigateTo`
- 需要返回上一级的页面：
  - 优先使用 `navigateBack`
  - 如果不存在上一级，则跳回约定入口页

## 16.1.1 手机导航与回退统一规则

为避免 Uni-app 客户端出现“系统返回可用，但页面自定义返回不可用”或“TabBar 页面无法正确回退”的问题，统一遵守以下规则：

- 所有非 TabBar 页面必须提供左上角返回按钮
- 左上角返回按钮行为必须与手机系统返回行为保持一致
- 非 TabBar 页面优先执行 `navigateBack`
- 当页面栈中不存在上一级时，必须按约定兜底跳转，不能出现返回后白屏或无响应
- TabBar 页面之间禁止使用 `navigateTo` 互跳，统一使用 `switchTab`
- 从 TabBar 页面进入非 TabBar 页面时，必须保证可通过系统返回回到来源 TabBar 页面
- 从搜索结果页进入攻略详情或 Vlog 详情时，系统返回应回到搜索结果页，而不是直接回首页或回 TabBar
- 从我的发布、我的收藏、我的点赞、他人主页进入详情时，系统返回应回到原来源页
- 发布成功、保存成功、匹配成功等“完成态跳转”页面，必须同时兼容页面返回按钮和手机系统返回
- 不允许依赖浏览器式历史行为做导航判断，必须基于 Uni-app 页面栈处理

## 16.1.2 导航调用矩阵

## 18. 搜索系统详细规则

## 18.1 搜索范围

### 首页搜索

搜索范围：

- 攻略
- Vlog / 帖子
- 目的地关键词
- 标签关键词

默认结果类型：

- `all`

### 攻略列表页搜索

搜索范围：

- 攻略标题
- 攻略摘要
- 攻略目的地
- 攻略标签

默认结果类型：

- `strategy`

### Vlog 列表页搜索

搜索范围：

- Vlog 标题
- Vlog 地点
- Vlog 标签
- Vlog 文案

默认结果类型：

- `vlog`

## 18.2 搜索结果页结构

搜索结果页采用统一页方案：

- 全部搜索入口统一进入 `pages/search-result/index`
- 通过 `type` 参数区分结果类型，不新增双页面方案
- 首页默认进入 `type=all`
- 攻略列表页默认进入 `type=strategy`
- Vlog 列表页默认进入 `type=vlog`

搜索结果页统一包含以下区块：

- 顶部搜索框
- 搜索类型切换 Tab
- 搜索结果数量
- 双列瀑布流结果列表
- 空态展示
- 历史搜索（无关键词时）

结果布局规则：

- 默认使用双列方块卡片流
- 卡片风格参考小红书搜索结果，而不是全宽列表
- 攻略与 Vlog 在 `all` 模式下混排展示
- 卡片封面优先，文字信息控制在 2 到 3 行内

Tab 类型：

- 全部
- 攻略
- Vlog

## 18.3 搜索交互规则

- 输入关键词后点击搜索，进入搜索结果页
- 空关键词不发请求，只展示历史搜索或热门搜索
- 同一关键词重复搜索，覆盖当前结果
- 切换 Tab 时保留关键词，仅切换结果类型
- 下拉刷新时重新按当前关键词和类型查询
- 触底时按当前关键词和类型加载下一页
- 搜索命中 Vlog 时，结果先展示在统一搜索结果页中
- 用户点击 Vlog 结果卡片后，进入 `pages/vlog-detail/index?id=xxx`
- 不允许通过搜索直接把页面切换到底部 `Vlog` Tab 并让该 Tab 承载搜索态

## 18.4 搜索历史规则

本地保存最近搜索记录，最多保留 10 条：

- 首页搜索与列表搜索共用历史
- 重复关键词移动到最前
- 支持单条删除
- 支持全部清空

## 18.5 空态规则

### 无关键词时

- 展示搜索历史
- 展示后端配置的热门搜索词

### 有关键词但无结果时

- 展示“未找到相关内容”
- 提供推荐关键词
- 提供返回列表或清空关键词按钮

## 17.6 搜索结果展示字段

### 攻略结果卡片

- 封面图
- 标题
- 目的地
- 标签
- 浏览 / 收藏数量
- 双列方块卡片展示
- 标题最多 2 行
- 封面比例优先 `3:4` 或接近方形

### Vlog 结果卡片

- 封面图
- 标题
- 地点
- 标签
- 点赞 / 收藏数量
- 播放量或点赞数量
- 双列方块卡片展示
- 标题最多 2 行
- 视频封面比例优先 `3:4` 或接近方形

## 19. 页面状态机

本节用于定义每个页面的标准状态流转，便于代码生成和统一处理。

## 18.1 通用状态定义

所有页面按需使用以下状态：

- `idle`：初始态
- `loading`：加载中
- `success`：加载成功
- `empty`：数据为空
- `error`：加载失败
- `submitting`：提交中
- `done`：提交完成

## 18.2 各页面状态机

### 登录页

- 初始：`idle`
- 提交登录：`submitting`
- 登录成功：跳转首页
- 登录失败：`error`

### 首页

- 初始：`loading`
- 拉取推荐成功：`success`
- 推荐为空：`empty`
- 拉取失败：`error`

### 攻略列表页

- 初始：`loading`
- 有数据：`success`
- 无数据：`empty`
- 失败：`error`
- 下拉刷新：`loading`
- 加载更多：附加 `loadingMore`
