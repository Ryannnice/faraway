## 2. 当前阶段目标

### 2.1 产品目标

`FARAWAY` 新应用第一阶段的产品目标是完成一个可用的“找搭子”闭环。

闭环定义如下：

1. 用户注册并登录
2. 用户填写找搭子条件并发起实时匹配
3. 系统在活动请求池中完成 1 对 1 候选匹配
4. 双方分别决定是否同意赴约
5. 双方都同意后生成见面信息
6. 双方各自可发送一次备注
7. 到见面当天结束后，本次流程自动结束

### 2.2 当前执行范围

当前实际执行范围为 `P0`，只实现以下模块：

1. 账号注册、登录、登出
2. 首页
3. AI 攻略生成与结果展示
4. 当前用户资料读取与更新
5. 实时找搭子输入页
6. 实时匹配状态页
7. 我的搭子页
8. 通知页
9. 他人主页基础展示
10. 设置页
11. 图片上传能力，当前仅用于头像

### 2.3 当前不做

当前阶段不实现以下内容：

1. 攻略内容系统
2. Vlog / 帖子系统
3. 搜索
4. 收藏 / 点赞 / 评论
5. AI 攻略草稿保存
6. AI 攻略正式发布
7. 首页搜索框
8. 首页 Preview 内容流
9. 首页悬浮发布入口
10. 聊天系统
11. 匹配历史页
12. 回访与处罚系统
13. 多人组局
14. 地图 SDK 深度接入
15. WebSocket / SSE
16. 管理后台
17. 旧招募帖公开流
18. 旧“我的招募帖 / 我的联系记录”页面

## 3. 核心产品定义

### 3.1 产品定位

新应用第一阶段是一个以“找搭子”为主链路的旅行匹配产品。

当前阶段不将内容社区作为主轴，不要求先恢复旧项目中的内容流能力。

### 3.2 新应用与旧项目的关系

新应用与旧项目关系如下：

1. 复用当前项目的 JWT、DashScope、OSS、统一响应格式、找搭子状态机规则
2. 不复用当前前端目录结构
3. 不复用当前后端单文件路由结构
4. 不复用当前 SQLite 单表 JSON 文档存储方案
5. 不要求迁移旧业务数据

### 3.3 关键实现决策

为保证可以直接开工，当前阶段采用以下固定决策：

1. 新应用不保留旧招募模型
2. 新应用不再使用“隐藏帖子”作为持久化中间层
3. 新应用的匹配池直接由活动中的 `match_requests` 构成
4. 当前阶段只实现账号密码登录
5. 手机验证码登录、微信登录延后
6. 当前阶段只支持图片上传，不实现视频上传
7. 当前阶段仅实现移动端客户端工程
8. 登录成功默认进入首页
9. P0 不做 TabBar，只做普通页面跳转
10. 首页保留当前视觉方向和两个主 CTA：`做攻略`、`找搭子`
11. 首页不保留搜索框、Preview 内容流、悬浮发布按钮
12. AI 攻略纳入 P0，但只做生成与展示，不做草稿保存和正式发布
13. 当前阶段所有业务时间统一按 `Asia/Shanghai` 处理
14. 所有 `match` 相关接口采用惰性状态推进
15. 登录接口仅返回 `token` 与 `userInfo`
16. `logout` 为无状态确认接口，不做 token 撤销
17. 通知页只读展示，不做已读能力

说明：

1. “隐藏帖子池”是旧系统为兼容公开招募流而引入的历史结构
2. 新应用 P0 已明确不实现公开招募流，因此该中间层不再保留
3. 匹配算法直接面向 `match_requests` 即可

## 4. 技术架构要求

### 4.1 仓库结构

新应用按单仓库组织，推荐结构如下：

```text
FARAWAY/
├── apps/
│   ├── api/
│   └── mobile/
├── docs/
├── infra/
└── scripts/
```

### 4.2 前端技术栈

`apps/mobile` 使用以下技术栈：

1. `Uni-app`
2. `Vue 3`
3. `TypeScript`
4. `Pinia`
5. `SCSS`

前端要求：

1. 页面统一使用 `script setup`
2. 页面层只处理展示与交互
3. 所有接口请求统一走 `api/`
4. 所有跨页共享状态统一进入 `stores/`
5. 不允许页面层直接拼接业务接口字段
6. 不允许继续使用当前项目中的轻量对象 store 写法

### 4.3 后端技术栈

`apps/api` 使用以下技术栈：

1. `FastAPI`
2. `SQLAlchemy 2.x`
3. `Pydantic v2`
4. `Alembic`
5. `PostgreSQL`

后端要求：

1. 按业务模块拆分代码
2. 路由、schema、service、repository、model 分层
3. 不允许将全部业务继续堆入单一 router 文件
4. 所有核心状态流转必须通过 service 层完成
5. 所有多实体更新操作必须具备事务边界

### 4.4 目录规范

4. 修改密码

1. 注册成功后创建用户资料默认值：
   - `nickname = username`

   - `avatar = ""`

   - `token`

4. `logout` 为无状态确认接口

## 5.5 实时找搭子模块

### 5.5.1 主链路

实时找搭子模块必须覆盖以下链路：

1. 进入找搭子页
2. 获取当前实时匹配状态
3. 无活动请求时展示输入页
4. 提交条件后创建活动请求
5. 活动请求进入匹配中状态
6. 匹配出候选对象后进入待决定状态
7. 单方同意时保持待决定状态
8. 单方拒绝时回到匹配中状态
9. 双方都同意后进入已确认赴约状态
10. 双方各自可提交一次备注
11. 到见面当天结束后进入完成态

### 5.5.2 输入字段

找搭子表单固定字段如下：

1. `destination`
2. `travel_start_date`
3. `travel_end_date`
4. `preference_tags`
5. `preference_text`

### 5.5.3 输入校验规则

1. `destination` 必填
2. `travel_start_date` 必填
3. `travel_end_date` 必填
4. `travel_end_date >= travel_start_date`
5. `preference_tags` 与 `preference_text` 至少有一项非空
6. `travel_start_date` 与 `travel_end_date` 统一使用 `YYYY-MM-DD` 日期字符串
7. 由于 P0 只存日期不存出发时刻，所有出行日期校验统一按 `Asia/Shanghai` 的自然日粒度处理
8. `travel_start_date` 最早只能为“当前自然日 + 3 天”

### 5.5.4 偏好标签枚举

当前阶段统一使用以下标签：

```json
[
  "特种兵",
  "慢旅行",
  "拍照",
  "美食",
  "自然风光",
  "人文历史",
  "早起",
  "夜景"
]
```

### 5.5.5 单用户活动请求规则

同一用户同一时刻仅允许存在一条活动中的 `match_request`。

活动状态包括：

1. `pending`
2. `matched_waiting_decision`
3. `matched_accepted`

规则如下：

1. 若当前为 `pending`，允许用户重新提交条件
2. 重新提交时取消旧请求并创建新请求
3. 若当前为 `matched_waiting_decision`，不允许覆盖
4. 若当前为 `matched_accepted`，不允许重新发起

### 5.5.6 页面视图要求

找搭子页只保留一个页面，按状态切换以下视图：

1. 输入页
2. `pending` 匹配中页
3. `matched_waiting_decision` 待决定页
4. `matched_accepted` 已确认赴约页
5. `failed / cancelled / finished` 结束页

### 5.5.7 匹配中状态要求

`pending` 状态下：

1. 页面展示当前输入条件
2. 页面每 5 秒轮询一次当前状态
3. 用户允许离开页面
4. 离开页面不自动取消请求
5. 再次进入时按后端状态恢复页面
6. 页面必须提供“取消匹配”入口

### 5.5.8 候选状态要求

`matched_waiting_decision` 状态下：

1. 页面必须展示候选对象
2. 页面主体必须展示同意与拒绝按钮
3. 首次进入状态时额外弹一次匹配弹窗
4. 关闭弹窗只关闭 UI，不改变后端状态
5. 再次进入页面时，如果候选仍有效，页面主体继续展示该候选
6. 已关闭过的弹窗不重复自动弹出

### 5.5.9 候选同意规则

1. 双方都同意后才生成 `match_pair`
2. 若 A 已同意、B 未同意，则请求状态仍为 `matched_waiting_decision`
3. 单方同意时，双方 `candidate.status` 仍保持 `pending_decision`
4. 单方同意时需保存：
   - `my_decision = accepted`
   - `peer_decision = pending`
5. 前端必须展示“等待对方回应”状态
6. 双方都同意后：
   - 双方 `candidate.status = accepted`
   - 双方 `match_request.status = matched_accepted`
   - 双方 `match_request.current_candidate_id` 清空
   - 双方 `match_request.current_pair_id` 写入生成的 `pair.id`

### 5.5.10 候选拒绝规则

1. 用户点击“拒绝，继续寻找”后，当前候选状态变为 `rejected`
2. 对应镜像候选也必须同步变为 `rejected`
3. 双方对应请求重新回到 `pending` 或 `failed`
4. 同一 `match_request` 中，被拒绝或过期过的同一对象不得重复推荐
5. 用户新建一条新的请求后，可再次匹配到同一对象

### 5.5.11 已确认赴约规则

`matched_accepted` 状态下：

1. 页面展示对方头像与昵称
2. 页面展示见面时间
3. 页面展示见面地点
4. 页面展示我的备注
5. 页面展示对方备注
6. 若我方尚未提交备注，则展示备注输入框和提交按钮
7. 一旦提交成功，输入区变为只读展示

### 5.5.12 备注规则

1. 每个用户只能发送一次备注
2. 备注提交后不可编辑
3. 备注提交成功后对方可见

### 5.5.13 完成规则

1. 匹配成功后，该流程持续展示到见面当天结束
2. 见面当天结束后，`match_pair.status = finished`
3. 对应 `match_request.status = finished`
4. 用户再次进入找搭子页时，若最近一条请求为 `failed / cancelled / finished`，页面先展示结束页
5. 用户点击“重新开始”后，再进入输入页并允许发起下一轮请求

### 5.5.14 惰性状态推进补充规则

1. 若 `match_request.status = pending` 且当前时间已超过 `match_deadline_at`，则推进为 `failed`
2. 若 `match_request.status = matched_waiting_decision` 且当前候选已 `rejected` 或 `expired`，则清空 `current_candidate_id`
3. 第 2 条发生后，若当前时间仍未超过 `match_deadline_at`，则请求回到 `pending`
4. 第 2 条发生后，若当前时间已超过 `match_deadline_at`，则请求推进为 `failed`
5. 若 `match_pair.status = active` 且当前时间已超过 `meet_time` 所在自然日的 `23:59:59 +08:00`，则 `match_pair` 与对应双方 `match_request` 一并推进为 `finished`
6. 所有状态推进必须在单个事务内完成

## 5.7 通知模块

### 5.7.1 当前阶段通知策略

当前阶段仅实现轻量通知记录，不实现复杂推送。

后端在以下时机写入通知：

1. 找到新的候选对象
2. 双方成功确认赴约

### 5.7.2 当前不做

1. 全局即时弹窗
2. 推送服务
3. 未读数复杂聚合
4. 标记已读能力

### 5.7.3 通知类型

P0 固定通知类型如下：

1. `realtime_match_found`
2. `realtime_match_confirmed`

对应 `payload` 规则如下：

1. `realtime_match_found` 包含 `requestId` 与 `candidateId`
2. `realtime_match_confirmed` 包含 `requestId` 与 `pairId`

## 5.8 图片上传模块

### 5.8.1 功能范围

当前阶段仅实现图片上传能力，用于：

1. 用户头像上传

### 5.8.2 上传方式

头像上传采用两步流程：

1. 前端先调用 `POST /api/upload/image`
2. 上传成功后拿到 `assetId` 与 `url`
3. 前端再调用 `PUT /api/user/profile` 写入 `avatar`

上传约束如下：

1. 当前阶段仅允许 `image/jpeg`、`image/png`、`image/webp`
2. 单张图片大小不超过 `10MB`
3. 成功上传后后端必须写入 `media_assets`

### 5.8.3 当前不做

1. 视频上传
2. 内容发布媒体上传

## 6. 匹配算法需求

### 6.1 匹配池定义

新应用中，匹配池由所有活动中的 `match_requests` 构成。

活动状态包括：

1. `pending`

### 6.2 候选基础条件

候选对象必须满足：

1. 不能是自己
2. 对方请求状态为 `pending`
3. 对方当前没有已确认的有效 `pair`
4. 地点规范化后完全一致
5. 日期范围至少有 1 天重叠
6. 对方不是当前请求已拒绝或已过期过的对象

地点规范化规则固定如下：

1. 去掉首尾空格
2. 将连续空格折叠为一个空格
3. 英文字母统一转小写
4. P0 不做地名别名词典，不做城市层级映射

### 6.3 排序规则

在满足基础条件后，按以下顺序排序：

1. 标签重合数量更多的优先
2. 自由文本关键词重合数量更多的优先
3. 日期重叠天数更多的优先
4. 更早进入池子的请求优先

### 6.4 自由文本关键词规则

当前阶段不做复杂 NLP。

最简规则如下：

1. 将 `preference_text` 按常见标点和空格拆词
2. 去掉空字符串
3. 统计双方词项重合数
4. 仅作为加分项，不作为硬性门槛

### 6.5 无候选时的处理

如果当前没有合适候选：

1. 请求保持 `pending`
2. 不伪造假匹配
3. 直到后续轮询时出现可匹配对象
4. 或直到达到截止时间变为 `failed`

## 7. 固定业务常量

为便于直接实现，当前阶段固定以下业务常量：

1. 匹配截止时间：`travel_start_date 当日 23:59:59 +08:00 - 48 小时`

## 8. 见面信息生成规则

### 8.1 见面时间

固定规则如下：

1. `meet_day` 取双方出行日期重叠区间的第一天

4. 当前阶段不允许用户修改
5. 当前阶段所有业务时间统一按 `Asia/Shanghai` 处理

### 8.2 见面地点

1. 白天

5. 易辨认

### 8.4 匹配摘要

`match_summary` 由后端生成，要求如下：

1. 中文单句

## 11.5 实时找搭子接口行为要求

### 11.5.1 通用状态推进规则

所有 `match` 相关接口必须先执行一次惰性状态推进。

适用接口包括：

1. `POST /api/match/realtime`
2. `GET /api/match/realtime/current`
3. `POST /api/match/realtime/candidate/{candidateId}/accept`
4. `POST /api/match/realtime/candidate/{candidateId}/reject`
5. `POST /api/match/realtime/pair/{pairId}/remark`
6. `POST /api/match/realtime/{requestId}/cancel`

推进规则如下：

1. 先同步请求、候选、pair 的过期与终态
2. `pending` 请求一旦达到 `match_deadline_at`，立即推进为 `failed`
3. `matched_waiting_decision` 候选一旦超过 `decision_expires_at`，对应双向候选置为 `expired`
4. 第 3 条发生后，对应双方请求按是否超过各自 `match_deadline_at` 分别推进为 `pending` 或 `failed`
5. `matched_accepted` 对应 `pair` 一旦超过 `meet_time` 所在自然日结束，立即推进为 `finished`
6. 所有推进必须在单个事务内完成
7. `POST /api/match/realtime` 与 `GET /api/match/realtime/current` 在状态同步后允许顺手尝试匹配

### 11.5.2 `POST /api/match/realtime`

请求体：

```json
{
  "destination": "大阪",
  "travel_start_date": "2026-05-10",
```

1. 若当前无活动请求，则新建请求
2. 若当前有 `pending` 请求，则取消旧请求并新建请求

4. 创建请求后允许立即尝试匹配
5. 成功返回 `status` 允许为：
   - `pending`
   - `matched_waiting_decision`

### 11.5.4 `POST /api/match/realtime/candidate/{candidateId}/accept`

行为要求：

1. 若仅我方已同意，则返回 `matched_waiting_decision`
2. 若双方都已同意，则生成 `pair` 并返回 `matched_accepted`
3. 仅允许操作当前用户自己的当前候选
4. 候选若已过期、已拒绝或已不处于 `pending_decision`，返回业务冲突
5. 单方同意时，双方 `candidate.status` 仍保持 `pending_decision`
6. 双方都同意后，双方 `candidate.status = accepted`
7. 双方都同意后，双方 `match_request.current_candidate_id` 必须清空
8. 双方都同意后，双方 `match_request.current_pair_id` 必须写入生成的 `pair.id`

返回规则：

1. `matched_waiting_decision` 时返回 `candidate`
2. `matched_accepted` 时返回 `pair`

### 11.5.5 `POST /api/match/realtime/candidate/{candidateId}/reject`

行为要求：

1. 当前候选标记为 `rejected`
2. 对应镜像候选也必须同步标记为 `rejected`
3. 对应双方请求回退为 `pending` 或 `failed`
4. 本接口不负责立即生成下一位候选，前端继续轮询当前状态即可

成功返回示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "status": "pending"
  }
}
```

### 11.5.6 `POST /api/match/realtime/pair/{pairId}/remark`

行为要求：

1. 当前用户仅可提交一次备注
2. 若已提交，再次提交返回业务冲突
3. `remark` 最大长度为 200

请求体：

```json
{
  "remark": "我会背黑色双肩包，提前 10 分钟到。"
}
```

成功返回时返回最新 `pair` 视图。

### 11.5.7 `POST /api/match/realtime/{requestId}/cancel`

行为要求：

1. 仅允许取消 `pending` 请求
2. 成功取消后状态变为 `cancelled`

成功返回示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "status": "cancelled"
  }
}
```

## 11.6 通知与上传接口行为要求

### 11.6.1 `GET /api/my/notifications`

行为要求：

1. 仅返回当前用户通知列表
2. 当前阶段只读展示，不做已读能力
3. 按 `createdAt` 倒序返回
4. 单次最多返回最近 50 条

成功返回示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "list": [
      {
        "id": "notice_001",
        "type": "realtime_match_found",
        "title": "搭子匹配",
        "content": "系统为你找到了一位新的候选搭子",
        "payload": {
          "requestId": "req_001",
          "candidateId": "cand_001"
        },
        "createdAt": "2026-04-29T12:00:00+08:00",
        "updatedAt": "2026-04-29T12:00:00+08:00"
      }
    ]
  }
}
```

### 11.6.2 `POST /api/upload/image`

行为要求：

1. 使用 multipart/form-data 上传图片
2. 表单字段名固定为 `file`
3. 成功返回至少包含：
   - `assetId`
   - `url`
4. 图片大小限制为 `10MB`
5. 当前阶段仅接受 `jpeg`、`png`、`webp`

成功返回示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "assetId": "asset_001",
    "url": "https://example.com/avatar.jpg"
  }
}
```

## 12. 非功能需求

## 12.1 安全

1. 密码必须安全哈希存储
2. 后端不得允许匿名访问受保护接口
3. 后端不得信任前端伪造用户身份头
4. 所有用户隔离必须以后端 token 解析出的用户 ID 为准

## 12.2 可维护性

1. 前后端必须使用明确类型定义
2. 不允许继续使用无边界的“大一统文件”
3. 关键业务常量需集中管理
4. 关键枚举需集中管理

## 12.3 可测试性

P0 必须具备自动化测试，至少覆盖：

1. 注册成功
2. 登录成功
3. 获取与更新当前用户资料成功
4. 头像上传成功并可写回资料
5. 创建实时请求成功
6. `pending` 请求被新请求覆盖
7. 两个用户成功匹配出候选
8. 单方同意后的等待状态
9. 双方都同意后生成 `pair`
10. 单方拒绝后双方回退
11. 备注只能提交一次
12. `GET /api/match/realtime/current` 能正确返回终态记录
13. 见面当天结束后状态自然结束

## 12.4 日志与调试

后端至少记录以下日志：

1. 登录成功与失败
2. 请求创建与取消
3. 候选生成
4. 同意与拒绝操作
5. `pair` 生成
6. AI 见面地点生成失败回退
7. OSS 上传失败

## 15. P0 验收标准

P0 完成必须满足以下标准：

1. 用户 A 可注册并登录
2. 用户 B 可注册并登录
3. 登录成功后默认进入首页
4. 首页仅保留 `做攻略` 与 `找搭子` 两个主入口
5. 用户可读取并更新自己的资料
6. 用户可上传头像并成功写回资料
7. 用户可查看他人主页基础信息
8. 用户可进入 AI 攻略页并成功生成攻略结果
9. AI 攻略结果可展示，但不会写入草稿和正式内容
10. A 创建实时匹配请求成功
11. B 创建实时匹配请求成功
12. 系统能将 A 与 B 匹配为候选
13. 通知页可看到候选匹配通知
14. A 先同意时，A 进入等待对方状态
15. B 后同意时，双方生成最终 `pair`
16. 通知页可看到匹配成功通知
17. 我的搭子页可看到当前状态摘要或已确认搭子摘要
18. 双方都能看到见面时间与见面地点
19. 双方都只能各提交一次备注
20. 设置页退出登录后，受保护页面需重新登录
21. 到见面当天结束后，流程自然结束
22. 用户再次进入找搭子页时，能重新发起下一轮请求
