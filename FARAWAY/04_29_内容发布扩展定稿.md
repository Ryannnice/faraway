# 04_29 FARAWAY 内容发布扩展定稿

## 1. 文档定位

### 1.1 目的

本文档用于在既有 `04_29_完整重构找搭子.md` 的 `P0` 基线之上，补充一轮新的产品与技术决策。

本轮新增目标不再局限于“AI 攻略生成与展示”，而是向内容发布闭环扩展，覆盖：

1. 真实 AI 攻略生成
2. AI 攻略编辑后正式发布
3. `Vlog` / 帖子发布
4. 首页公开内容流恢复
5. 内容详情页
6. 我的内容页

### 1.2 与旧文档的关系

本文档是增量定稿，不替代整份 `04_29_完整重构找搭子.md`，但当以下主题发生冲突时，以本文档为准：

1. `AI 攻略是否只做展示`

3. `是否实现 Vlog / 帖子系统`
4. `首页是否恢复内容流与发布入口`

说明：

1. 原 `P0` 文档仍作为找搭子主链路、认证、资料、通知、上传约束的基线
2. 本文档只覆盖本轮已确认扩展范围
3. 未在本文档重新定义的条目，继续沿用原文档

## 2. 本轮执行范围

### 2.1 新增执行范围

在原 `P0` 基线之上，本轮新增以下执行范围：

1. AI 攻略必须走真实 DashScope / Qwen 调用链路
2. AI 攻略生成结果可在前端编辑后正式发布
3. 新增图文 `Vlog` / 帖子发布能力
4. 首页恢复公开内容流
5. 首页恢复悬浮发布入口
6. 新增内容详情页
7. 新增我的内容页

### 2.2 当前不做

本轮依然不做以下能力：

1. AI 攻略草稿保存
2. 自动草稿恢复
3. 真视频上传
4. 真视频播放
5. 搜索
6. 收藏 / 点赞 / 评论
7. 内容审核后台
8. 内容推荐排序
9. 首页复杂搜索框
10. 首页多路推荐分栏

## 3. 关键产品决策

### 3.1 AI 攻略

本轮固定如下：

1. `AI 攻略` 继续使用 DashScope / Qwen 链路
2. 不再接受“本地模板伪装 AI 结果”作为默认成功路径
3. 若 AI 配置缺失、上游失败、超时或返回坏结构，接口直接失败
4. 前端必须明确向用户展示失败提示
5. 用户可重新生成
6. 自动化测试不得依赖本地模板回退，统一通过 mock DashScope 上游完成

### 3.2 攻略发布

本轮固定如下：

1. AI 攻略生成成功后，用户可直接进入发布态
2. 发布前允许编辑
3. 用户只有两个选择：
   - 放弃离开，不保存
   - 一键发布，直接生成正式内容
4. 不引入草稿态
5. 不引入“待审核”中间态
6. 点击正式发布前必须弹出二次确认
7. 当前阶段不实现“已发布后编辑”与“已发布后删除”

### 3.3 Vlog 定义

本轮 `Vlog` 固定定义为：

1. 图文帖子
2. 支持标题、正文、图片、标签
3. 不支持视频文件
4. 不支持视频封面、时长、播放器等视频特有能力

补充约定：

1. `vlog` 仅作为内部 `content_type`
2. 前端用户可见文案统一使用 `图文`

### 3.4 首页内容流

本轮首页固定如下：

1. 保留当前首页视觉方向与两个主 CTA：`做攻略`、`找搭子`
2. 在 Hero 区域下方恢复公开内容流
3. 内容流采用混排模式，同时展示：
   - 已发布攻略
   - 已发布图文 Vlog / 帖子
4. 每条内容卡片必须显示标签
5. 首页恢复悬浮发布入口

补充约定：

1. 这里的“公开”定义为：对所有已登录用户公开
2. 当前阶段不提供匿名访问首页内容流
3. 当前阶段不提供匿名访问内容详情页

### 3.5 我的内容

本轮新增：

1. 用户可查看自己已发布的内容列表
2. 列表至少包含攻略和图文 Vlog 两类内容
3. 当前阶段不做复杂筛选和统计
4. 当前阶段不做回收站、删除和恢复

## 4. 页面与交互要求

### 4.1 首页

首页必须同时承担以下职责：

1. 登录成功后的默认落点
2. 保留 `做攻略` 与 `找搭子` 两个主入口
3. 展示站内公开内容流
4. 提供悬浮发布入口

首页内容流卡片至少展示：

1. 内容类型标签：`攻略` 或 `图文`
2. 业务标签列表
3. 标题
4. 封面图或首图
5. 概要文案
6. 作者昵称与头像
7. 发布时间

列表加载规则：

1. 首页内容流采用分页加载
2. 分页从 `page=1` 开始
3. 默认 `page_size=20`
4. `page_size` 最大值为 `50`
5. 排序固定为 `published_at DESC, id DESC`

### 4.2 AI 攻略页

AI 攻略页分为两个阶段：

1. 输入生成条件阶段
2. 结果展示与发布阶段

结果展示与发布阶段要求：

1. 继续展示原有结构化结果：
   - `destination`
   - `overview`
   - `daily_plans`
   - `tips`
2. 新增可编辑发布字段：
   - `title`
   - `tags`
   - `cover_image`
3. `overview`、`daily_plans`、`tips` 允许用户修改后再发布
4. 提供 `重新生成`、`放弃`、`发布攻略` 三个动作

发布字段约束：

1. `title` 必填，长度 `1-60`
2. `tags` 最多 `5` 个
3. 单个业务标签长度 `1-12`
4. `cover_image` 选填；若填写，必须来自上传接口返回的 `asset_id`
5. `overview` 发布时不能为空，最大长度 `1000`
6. `daily_plans.length` 必须仍等于 `days`
7. 每个 `daily_plans[i]` 仍必须包含：
   - `day`
   - `activities`
   - `food`
   - `accommodation`
8. `tips` 允许为空数组，若非空则最多 `10` 条
9. 点击 `发布攻略` 后必须再次确认，确认后才真正写入数据库

字段映射约定：

1. 页面层可使用 `cover_image` 作为表单字段名
2. 调用发布接口时，前端必须将其映射为 `cover_asset_id`

### 4.3 图文 Vlog 发布页

新增独立图文发布页，至少包含：

1. `title`
2. `content`
3. `tags`
4. `images`

规则如下：

1. `title` 必填
2. `content` 必填
3. `tags` 选填
4. `images` 选填，但只允许图片
5. 当前阶段不要求必须有图片
6. `title` 长度 `1-60`
7. `content` 长度 `1-5000`
8. `tags` 最多 `5` 个
9. 单个业务标签长度 `1-12`
10. `images` 最多 `9` 张
11. 点击正式发布前必须再次确认

### 4.4 内容详情页

新增统一内容详情页。

详情页必须兼容两类内容：

1. 攻略
2. 图文 Vlog

攻略详情至少展示：

1. 标题
2. 封面图
3. 标签
4. 作者信息
5. 概述
6. 每日行程
7. Tips
8. 发布时间

图文 Vlog 详情至少展示：

1. 标题
2. 图片列表
3. 标签
4. 作者信息
5. 正文
6. 发布时间

### 4.5 我的内容页

新增我的内容页，至少展示：

1. 内容类型
2. 标题
3. 标签
4. 发布时间
5. 封面图或首图

当前阶段不要求：

1. 编辑已发布内容
2. 删除内容
3. 草稿列表

### 4.6 发布入口

发布入口固定如下：

1. 首页右下角新增悬浮发布按钮
2. 点击后弹出动作选择
3. 当前阶段提供两个动作：
   - `发布攻略`
   - `发图文`
4. AI 攻略页在生成成功后也必须提供直接发布入口

## 5. 数据与模型要求

### 5.1 内容类型

当前阶段只定义两种发布内容：

1. `strategy`
2. `vlog`

补充约定：

1. `strategy` 与 `vlog` 是稳定后端枚举值
2. 前端展示文案分别为 `攻略` 与 `图文`

### 5.2 建议存储模型

为保持实现简单，本轮内容系统采用：

1. 一张内容主表
2. 一张内容媒体关联表

建议主表字段至少包含：

1. `id`
2. `user_id`
3. `content_type`
4. `source_type`
5. `title`
6. `summary`
7. `cover_asset_id`
8. `cover_url`
9. `tags_json`
10. `content_payload`
11. `published_at`
12. `created_at`
13. `updated_at`

说明：

1. `content_payload` 允许按内容类型保存结构化正文
2. 攻略与 Vlog 不额外拆两套主表
3. 不回退到旧项目的公开招募帖结构
4. `source_type` 至少支持：
   - `ai_strategy`
   - `manual_vlog`
5. 当前阶段不保存原始 AI prompt 与原始 AI 返回全文

### 5.3 媒体关系

内容发布媒体继续复用已有 `media_assets`。

规则如下：

1. 内容中引用的图片必须来自上传接口返回的 `asset_id`
2. 发布时必须校验该图片确实属于当前用户
3. Vlog 多图按顺序保存
4. 攻略封面图允许为空
5. 图片上传继续复用 `POST /api/upload/image`
6. 当前阶段不新增独立内容图片上传接口

建议内容媒体关联表字段至少包含：

1. `id`
2. `content_id`
3. `asset_id`
4. `sort_order`
5. `created_at`

### 5.4 标签规则

当前阶段固定如下：

1. 每条内容都有一个由 `content_type` 派生的系统展示标签：
   - `攻略`
   - `图文`
2. 用户可额外填写业务标签
3. 数据库存储只保存业务标签，不单独落库系统展示标签
4. 接口返回给前端时，`tags` 字段按以下顺序拼接：
   - 系统展示标签
   - 用户业务标签
5. 业务标签按字符串数组存储
6. 后端发布前需要做：
   - 去首尾空格
   - 去重
   - 过滤空字符串
7. 前端调用发布接口时，`tags` 只传业务标签，不传系统展示标签

### 5.5 摘要与封面生成规则

为保证首页内容流和我的内容页展示稳定，固定如下：

1. `strategy.summary` 默认取发布时 `overview`，最大展示长度 `120`
2. `vlog.summary` 默认取正文前 `120` 个字符
3. `strategy.cover_url` 取用户选择的封面图；未选择则为空字符串
4. `vlog.cover_url` 优先取第一张图片；没有图片则为空字符串
5. 详情页展示全文，列表页只使用 `summary`

### 5.6 `content_payload` 结构

`strategy` 类型的 `content_payload` 至少包含：

1. `destination`
2. `days`
3. `overview`
4. `daily_plans`
5. `tips`

`vlog` 类型的 `content_payload` 至少包含：

1. `content`
2. `image_asset_ids`
3. `image_urls`

### 5.7 作者对象

涉及内容列表与详情时，`author` 字段固定至少包含：

1. `user_id`
2. `nickname`
3. `avatar`

## 6. 接口要求

### 6.1 AI 攻略

保留：

1. `POST /api/ai/generate-strategy`

行为要求补充如下：

1. 必须真实调用 DashScope / Qwen
2. 不再默认返回本地模板结果
3. 返回前仍需规整为固定 JSON 结构
4. 上游失败时返回失败，不写入数据库
5. `DASHSCOPE_API_KEY` 缺失时返回失败
6. 自动化测试统一通过 mock 上游覆盖成功与失败分支

### 6.2 内容发布

新增以下接口：

1. `POST /api/content/strategy/publish`
2. `POST /api/content/vlog/publish`
3. `GET /api/content/feed`
4. `GET /api/content/{content_id}`
5. `GET /api/my/contents`

鉴权规则固定如下：

1. 上述所有内容接口都要求登录
2. 当前阶段不新增匿名读取接口

### 6.3 `POST /api/content/strategy/publish`

职责：

1. 接收 AI 结果编辑后的正式发布数据
2. 保存为 `strategy` 类型内容
3. 成功后返回内容摘要与 `content_id`

请求体至少包含：

1. `title`
2. `tags`
3. `cover_asset_id`
4. `destination`
5. `days`
6. `overview`
7. `daily_plans`
8. `tips`

补充约定：

1. `cover_asset_id` 允许为 `null` 或空字符串
2. 成功返回体至少包含：
   - `content_id`
   - `content_type`
   - `title`
   - `summary`
   - `cover_url`
   - `tags`
   - `author`
   - `published_at`

### 6.4 `POST /api/content/vlog/publish`

职责：

1. 接收图文帖子发布数据
2. 保存为 `vlog` 类型内容
3. 成功后返回内容摘要与 `content_id`

请求体至少包含：

1. `title`
2. `content`
3. `tags`
4. `image_asset_ids`

成功返回体至少包含：

1. `content_id`
2. `content_type`
3. `title`
4. `summary`
5. `cover_url`
6. `tags`
7. `author`
8. `published_at`

### 6.5 `GET /api/content/feed`

职责：

1. 返回首页公开内容流
2. 按发布时间倒序
3. 混排攻略与 Vlog

返回项至少包含：

1. `content_id`
2. `content_type`
3. `title`
4. `summary`
5. `cover_url`
6. `tags`
7. `author`
8. `published_at`

分页协议固定如下：

1. 查询参数使用 `page` 与 `page_size`
2. 返回结构至少包含：
   - `list`
   - `total`
   - `page`
   - `page_size`
   - `has_more`
3. `list` 中每项字段与发布成功返回摘要保持一致

### 6.6 `GET /api/content/{content_id}`

职责：

1. 返回单条内容详情
2. 按内容类型返回结构化详情体

返回要求：

1. 返回公共摘要字段
2. 返回类型专属正文数据
3. 若内容不存在，返回 `404`
4. 公共摘要字段至少包含：
   - `content_id`
   - `content_type`
   - `title`
   - `summary`
   - `cover_url`
   - `tags`
   - `author`
   - `published_at`
5. `strategy` 详情正文字段至少包含：
   - `destination`
   - `days`
   - `overview`
   - `daily_plans`
   - `tips`
6. `vlog` 详情正文字段至少包含：
   - `content`
   - `image_urls`

### 6.7 `GET /api/my/contents`

职责：

1. 返回当前用户自己的已发布内容
2. 默认按发布时间倒序
3. 分页协议与 `GET /api/content/feed` 保持一致
4. 返回项字段与 `GET /api/content/feed` 保持一致

## 8. 验收标准

本轮扩展完成后，至少满足以下标准：

1. 配置正确时，`POST /api/ai/generate-strategy` 走真实 DashScope / Qwen
2. DashScope 缺配置或失败时，前后端能明确报错，不再返回伪 AI 成功结果
3. 用户可生成 AI 攻略并看到结构化结果
4. 用户可编辑 AI 攻略并正式发布
5. 用户放弃时，不产生草稿记录
6. 用户可发布图文 Vlog / 帖子
7. 首页可混排展示攻略与 Vlog
8. 首页卡片可显示标签
9. 点击首页内容卡片可进入详情页
10. 我的内容页可看到自己已发布的攻略与 Vlog
11. 发布内容引用的图片都必须来自当前用户自己的上传记录
12. 整个发布链路仍然不依赖视频上传
13. 首页内容流与我的内容页均按统一分页协议返回
14. 对外展示文案使用 `攻略` 与 `图文`，不要求在前端直接显示英文 `vlog`

---

## 9. 实现顺序建议

为降低回归风险，本轮建议按以下顺序实现：

1. 去掉 AI 本地模板默认成功回退，接通真实 DashScope / Qwen
2. 新增内容主表与内容媒体关联表
3. 完成攻略发布接口
4. 完成图文 Vlog 发布接口
5. 完成首页混排内容流接口与详情接口
6. 完成我的内容接口
7. 完成前端首页、攻略页、内容详情页、我的内容页、Vlog 发布页联调

说明：

1. 不先做评论、点赞、收藏
2. 不先做已发布内容编辑
3. 不先做真视频能力
